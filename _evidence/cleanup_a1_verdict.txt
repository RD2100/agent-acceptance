1. A1 中不要真正实现完整 Pre-GPT

你现在写的 Layer 2 是最终架构的一部分，但 A1 的范围最好仍然控制住。

这些放到 A2 更合适。

2. current.json missing 的语义要区分场景

你写的是：

Pre-Task: current.json missing/stale → WARNING
Pre-GPT: current.json missing → BLOCKED

这个是对的，但建议更精确：

Pre-Task:
  current.json missing → WARNING / INIT_REQUIRED
  current.json stale → SUGGEST_HANDOFF

Pre-GPT:
  current.json missing and cannot refresh → BLOCKED
  current.json missing but wrapper can initialize/refresh → allowed after writing evidence

否则第一次使用新项目时，Pre-GPT 可能被永久卡住。

3. metrics_source 要影响 force_handoff 的可信度

你已经写了：

metrics_source: cdp_dom_count > wrapper_counter > manual_estimate

建议再补一条规则：

manual_estimate should not trigger FORCE_HANDOFF by itself unless corroborated by another force condition.

也就是说：

assistant_message_count >= 60 + metrics_source=cdp_dom_count → FORCE
assistant_message_count >= 60 + metrics_source=wrapper_counter → FORCE
assistant_message_count >= 60 + metrics_source=manual_estimate → SUGGEST 或 NEEDS_REFRESH

否则 agent 手工估算一个数字，就可能触发强制 handoff，这个证据可信度不足。

4. auth_required 不应归入普通 handoff

你的 last_nav_result 枚举值合理：

ok
access_denied
not_found
timeout
auth_required
cdp_unavailable
unknown

建议增加处理规则：

access_denied / not_found → FORCE_HANDOFF
auth_required → HUMAN_REQUIRED
timeout / cdp_unavailable → SUGGEST_HANDOFF or RETRY_REQUIRED
unknown → WARNING

其中 auth_required 很重要。它不是“对话太长需要切换”，而是认证状态异常，应该进入 human_required，避免 agent 误判为换新对话即可解决。

5. Evidence Pack 要记录“FORCE 后是否继续执行”

Layer 3 里建议加一条硬规则：

decision == FORCE_HANDOFF and task continued without handoff or migration record → rejected

也就是说 evidence pack 不只是包含：

conversation-health/latest.json

还要在 review.yaml 中体现：

YAML
conversation_health:
  checked: true
  decision: FORCE_HANDOFF | SUGGEST_HANDOFF | OK | UNKNOWN | HUMAN_REQUIRED
  action_taken: continued | handoff_generated | migration_record_created | blocked
  justification: ...

这样 reviewer 可以判断：
即便触发了 FORCE，agent 有没有正确停止、handoff 或 migration。

6. A1 必须要求 negative-path evidence

既然这是一个 gate 任务，验收时不能只看单元测试。

A1 的 acceptance criteria 里建议明确要求 runtime negative-path evidence：

1. current.json missing → WARNING / INIT_REQUIRED
2. current.json stale → SUGGEST_HANDOFF
3. assistant_message_count >= 60 → FORCE_HANDOFF
4. review_round_count >= 3 → FORCE_HANDOFF
5. access_denied → FORCE_HANDOFF
6. auth_required → HUMAN_REQUIRED
7. invalid current.json → validator error
8. evidence pack missing conversation-health/latest.json → evidence incomplete
9. FORCE_HANDOFF but no handoff/migration record → rejected or blocked

这能延续前一个任务里已经形成的 evidence-first 标准。

可以将最终共识定稿为：

CONVERSATION-HEALTH-GATE adopts a four-layer defense model:

1. Pre-Task Hard Gate:
   sadp_pre_task_enforcer.py reads .ai/conversation/current.json only.
   FORCE_HANDOFF blocks task start.
   Missing or stale metrics produce WARNING/SUGGEST, not immediate BLOCKED.

2. Pre-GPT Gate:
   scripts/pre_gpt_gate.py refreshes runtime metrics during GPT/CDP submission.
   Legacy scripts temporarily import check_pre_gpt_gate().
   submit_gpt_review.py is the target unified wrapper.
   Missing current.json must be initialized or refreshed before submission; failure to refresh blocks submission.

3. Evidence Pack Hard Requirement:
   build_evidence_pack.py includes conversation-health/latest.json and current snapshot.
   review.yaml records conversation_health.
   Missing evidence or unresolved FORCE_HANDOFF prevents ACCEPTED verdict.

4. Pre-Commit Advisory:
   pre-commit.governance.ps1 adds conversation-health as ADVISORY only.
   It audits and warns but does not become the primary enforcement gate.

Thresholds are policy-driven via configs/conversation-health-policy.yaml.
Pre-Task uses last known state.
Pre-GPT/CDP wrapper performs real-time metrics capture.
chat_url accessibility is passively recorded through submit/capture failures.
Migration records are required when conversation switches occur.
