3. 仍然存在的关键问题
BLOCKER 1：Test-Governance 被降级为 advisory，偏离原任务验收标准

你最初的任务要求是：

YAML
test_governance_blocking_failure: blocks_commit

但 V2 实际改成了：

YAML
test_governance:
  blocking: false
  behavior: advisory_only

文档、hook 逻辑和测试都同步改成：

test-governance failure → PASS advisory

如果项目最终决定 Test-Governance 只做 advisory，这可以作为一个新的治理设计选择；但它不能在本任务中悄悄替代原验收标准。需要明确的 decision record：

YAML
required_decision:
  question: Should Test-Governance be blocking or advisory?
  current_v2_behavior: advisory
  previous_task_requirement: blocking
  reviewer_status: requires explicit governance decision

所以这一点仍是 blocker。

BLOCKER 2：validator 仍不够 fail-closed

validate_hook_output.py 虽然修掉了 PASS_WITH_WARNINGS，但仍有两个问题：

第一，它用默认值 0 处理缺失 stage：

Python
运行
exit_code = stages.get(stage_name, 0)

如果 sadp-audit 或 ai-guard 缺失，validator 可能把它当作通过。这不符合“missing required stage blocks”的要求。

第二，它允许所有 blocking stages 为 0 时 overall_result 是 BLOCKED：

Python
运行
if blocking_zeros and result not in ("PASS", "BLOCKED"):

这会让“无失败但 BLOCKED”的语义异常通过。更严格的语义应是：

YAML
if sadp_audit == 0 and ai_guard == 0:
  expected_overall_result: PASS

除非另有明确的 external block reason 字段。

因此，不能说 V2 已经证明了 runtime negative path。

BLOCKER 4：ai_guard exit code 仍有 fail-open 风险

V2 去掉了文本匹配，这是进步。但当前逻辑是：

PowerShell
$script:aiGuardJobExitCode = if ($null -ne $rawExit) { [int]$rawExit } else { 0 }

如果 PowerShell Job 没有可靠提供 ExitCode，它会默认通过。这是 fail-open。

更安全的策略应是：

PowerShell
if ($null -eq $rawExit) {
  $script:aiGuardJobExitCode = 1
  # record reason: AI_GUARD_EXIT_CODE_UNAVAILABLE
}

或者在 Job 内部显式运行 python 后捕获 $LASTEXITCODE 并返回结构化结果。
