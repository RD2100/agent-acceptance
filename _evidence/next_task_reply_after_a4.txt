确认：下一步应做 EVIDENCE-CAPTURE-STANDARD-A1。

原因是：UAWS-A1 已经完成，但它的 limitation 是“文档/治理标准有了，runtime enforcement 不足”；而最近 A3、A4 的 limitation 也都集中在 evidence pack 层，比如缺少 review.yaml / final-report.md / full regression log，或 runtime artifact 覆盖不足。所以现在最该补的不是再写一份规范，而是把 evidence pack 的必备结构、缺失处理、runtime evidence 要求、verdict eligibility 固化成可执行标准。

1. 下一步任务判断

优先级建议如下：

P0: EVIDENCE-CAPTURE-STANDARD-A1
P1: Conversation Registry Lifecycle / Migration Record Enforcement
P1: Universal Agent Workflow Standard runtime enforcement follow-up
P2: OPERATIONS-MANUAL-AUDIT-PATTERN-BIND-A1
P2: A3/A4 limitations cleanup

其中 EVIDENCE-CAPTURE-STANDARD-A1 是当前最优先。它可以直接吸收 A3/A4 的经验，避免后续每个证据包都反复出现“材料有，但不完整”“测试声称全量，但日志不在包里”“runtime evidence 只靠测试模拟”等问题。

2. EVIDENCE-CAPTURE-STANDARD-A1 任务描述

下面这份可以作为正式 TaskSpec 基础：

编辑

Task: EVIDENCE-CAPTURE-STANDARD-A1

Goal:
Create and enforce a canonical Evidence Capture Standard for all future agent tasks, so that evidence packs are complete, reviewable, schema-aware, and resistant to missing-evidence or stale-evidence acceptance.

Context:
UNIVERSAL-AGENT-WORKFLOW-STANDARD-A1 has already been completed with accepted_with_limitation. Its main limitation is that it is documentation/governance-oriented and does not provide runtime enforcement.

CONVERSATION-HEALTH-GATE-A1 through A4 established a multi-layer governance pattern, but A3 and A4 exposed recurring evidence-pack limitations:

missing review.yaml / final-report.md / full regression logs in some review packs;

runtime behavior sometimes proved by tests but not by standalone runtime artifacts;

claimed full regression sometimes not independently verifiable from the ZIP;

accepted_with_limitation decisions need consistent representation.

This task converts those lessons into a reusable Evidence Capture Standard.

Required Work:

Define the canonical evidence pack contract

Create or update:

docs/agent-runtime/evidence-capture-standard.md

docs/agent-runtime/evidence-pack-review-rules.md

Define required evidence categories:

task metadata

commit chain

diff / git-show

tests

runtime evidence

negative-path evidence

safety / secret scan

git status before/after

review.yaml

final-report.md

limitations and verdict eligibility

Standardize required files
Every standard evidence pack should include, unless explicitly marked not applicable:

review.yaml

final-report.md

git-status-before.txt

git-status-after.txt

test-output.txt

diff.patch

git-show.txt

safety-report.json

secret-scan output

task spec or current-task snapshot

evidence manifest

runtime evidence index when runtime behavior is claimed

Add evidence manifest support

Add a machine-readable manifest, for example:

evidence-manifest.json

It should record:

task_id

run_id

base_commit

head_commit

commit_chain

required_files_present

optional_files_present

runtime_evidence_files

negative_path_scenarios

test_summary

git_status_summary

verdict_eligibility

limitations

Strengthen build_evidence_pack.py

Update scripts/build_evidence_pack.py to validate the evidence pack contract.

Missing required files should not be silently ignored.

Missing required evidence should set:

verdict_eligibility: needs_more_evidence

In strict mode, missing required evidence may fail the build.

The builder must distinguish:

required

optional

not_applicable

missing_but_limitation

missing_blocking

Standardize runtime and negative-path evidence

Define a consistent format for runtime scenario files:

scenario name

expected result

actual result

status

source command or script

timestamp

Ensure review.md / final-report.md tables are generated from scenario files rather than hardcoded assumptions.

Prevent stale runtime evidence from being mixed into current evidence packs.

Standardize full regression claims

If a report claims “full regression passed”, the full regression log must be included.

If only task-specific tests were run, the report must say so explicitly.

review.yaml must distinguish:

full_regression

targeted_tests

runtime_scenarios

skipped_tests_with_rationale

Standardize verdict eligibility

Define when a pack is eligible for:

ACCEPTED

ACCEPTED_WITH_LIMITATION

NEEDS_REVISION

REJECTED

Required rule:

A pack missing review.yaml or final-report.md cannot be ACCEPTED.

A pack claiming runtime behavior without runtime evidence cannot be ACCEPTED, unless explicitly accepted_with_limitation.

modified_tracked must be 0 for a clean accepted baseline unless a deferred decision record exists.

blocked / partial / human_required must never be represented as success.

Add tests
Add tests covering:

complete evidence pack passes validation;

missing review.yaml marks needs_more_evidence;

missing final-report.md marks needs_more_evidence;

missing full regression log downgrades full regression claim;

runtime claim without runtime artifact creates limitation or needs_more_evidence;

stale extra evidence is rejected or excluded;

modified_tracked > 0 prevents clean accepted eligibility;

accepted_with_limitation is represented distinctly from accepted.

Provide runtime evidence
Generate runtime evidence for:

complete_pack_valid

missing_review_yaml_needs_more_evidence

missing_final_report_needs_more_evidence

missing_runtime_artifact_downgrades

stale_extra_evidence_detected

modified_tracked_blocks_clean_acceptance

full_regression_claim_requires_log

Out of Scope:

Do not rewrite old accepted evidence packs.

Do not change Conversation Health Gate A1-A4 semantics.

Do not implement live dispatch enforcement.

Do not build a large new framework.

Do not replace all existing review scripts.

Do not make reviewer verdicts automatic; this task standardizes evidence eligibility, not final human/GPT judgment.

Acceptance Criteria:

Evidence Capture Standard documentation exists and is usable by future agents.

build_evidence_pack.py enforces or records required evidence completeness.

evidence-manifest.json or equivalent machine-readable manifest is generated.

review.yaml includes verdict_eligibility and evidence completeness fields.

Runtime scenario tables in final-report.md are generated from actual scenario files, not hardcoded assumptions.

Claims of full regression require included full regression logs.

Tests cover positive and negative evidence-pack validation paths.

Runtime negative-path evidence is included.

modified_tracked is 0 after commit.

Existing accepted Conversation Health Gate behavior does not regress.

3. 为什么不是先做 A3/A4 cleanup？

A3/A4 的 limitation 是非阻塞的，而且它们暴露的问题本质上不是某一个任务的小瑕疵，而是证据包标准还没有统一。

所以不建议单独做：

CONVERSATION-HEALTH-GATE-A3-CLEANUP
CONVERSATION-HEALTH-GATE-A4-CLEANUP

更好的方式是把这些 limitation 抽象进 EVIDENCE-CAPTURE-STANDARD-A1，以后所有证据包都按同一套规则生成和审核。

4. 其他任务优先级说明
P1：Conversation Registry Lifecycle / Migration Record Enforcement

这个也重要，尤其是你之前提到过：

旧对话无法访问 → agent 直接换新对话 → 未更新 registry / 未生成 migration record

但它依赖更稳定的 evidence pack 标准。否则 migration record 的证据也会继续出现“有文件但不完整”的问题。

所以它排在 EVIDENCE-CAPTURE-STANDARD-A1 之后更合适。

P1：UAWS runtime enforcement follow-up

UAWS-A1 的 limitation 是“没有 runtime enforcement”。
EVIDENCE-CAPTURE-STANDARD-A1 实际上就是它的第一个 runtime enforcement follow-up，因此不需要另开一个更高优先级的 UAWS 任务。

P2：OPERATIONS-MANUAL-AUDIT-PATTERN-BIND-A1

这个任务有价值，但现在不应抢占。它更适合在 Evidence Capture Standard 形成后做，用来把标准写回 operations manual，避免后续 agent 不知道现有工具。

5. 推荐执行顺序
1. EVIDENCE-CAPTURE-STANDARD-A1
   统一并强化 evidence pack 标准。

2. Conversation Registry Lifecycle / Migration Record Enforcement
   把 conversation migration 与 registry 状态变更纳入证据闭环。

3. UAWS runtime enforcement follow-up
   将 evidence capture、conversation health、startup/pre-task/pre-gpt/pre-commit 门禁统一纳入可执行 gate。

4. OPERATIONS-MANUAL-AUDIT-PATTERN-BIND-A1
   把以上模式绑定进 operations manual 和 audit pattern。

一句话：下一步应做 EVIDENCE-CAPTURE-STANDARD-A1，因为它正好把 UAWS-A1 的“文档标准”推进为“证据包可执行标准”，也能吸收 A3/A4 暴露出的 evidence pack limitation。
