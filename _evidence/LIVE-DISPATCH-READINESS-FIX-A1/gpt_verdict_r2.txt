三、目前还没有完成什么？
1. live dispatch 还没有授权

这是最重要的未完成项。

目前所有 review 都明确保留：

YAML
live_dispatch: NOT_AUTHORIZED

也就是说：

可以 dry-run；

可以做 readiness review；

可以检查 registry / binding / router；

但不能直接执行真实 live dispatch。

下一步应该做的是：

YAML
task_id: LIVE-DISPATCH-READINESS-REVIEW-A1
type: readiness_review_only
live_dispatch: NOT_AUTHORIZED

这个任务只审查“是否具备申请 human authorization 的条件”，不是直接 live。

2. 工作区仍不是 fully clean

现在比较好的地方是：

YAML
modified_tracked: 0

说明已跟踪文件没有脏改动。

但仍有 untracked entries。最新 R5 中仍是 registered closure，也就是说：

YAML
workspace_state: registered_closure
含义:
  modified_tracked: 0
  untracked: 有，但全部登记

这不是 blocker，但也不能说完全干净。

简单理解：

没有隐藏脏改动，但还有登记过的遗留文件。

3. NEG-009 和 secret-scan 文件还需要制度化

项目里一直反复出现：

NEG-009 mock secret fixtures
secret-scan-output.txt
deny_paths
deny_list

这些文件不是普通垃圾，而是负向测试和 mock secret 相关材料。

目前它们靠 deferred register 维持秩序，但还缺一个正式政策：

YAML
task_id: NEG-009-MOCK-SECRET-FIXTURE-POLICY-A1

这个任务要明确：

哪些 mock secret 文件可以存在
是否允许提交
应该放在哪个目录
如何避免被误认为真实 secret
secret-scan 输出如何保存或屏蔽
4. hook failure semantics 仍需最终收口

之前做过 hook 失败语义验证，但还没有形成最终 accepted 的稳定结论。

关键问题是：

sadp-audit 失败是否一定阻断
ai_guard 失败是否一定阻断
Test-Governance 是 blocking 还是 advisory
validator 对缺失 stage 是否 fail-closed
负向运行证据是否完整入包

这个问题不阻塞当前 evidence standard R5 的接受，但如果要进入 live readiness，就必须认真对待。

后续可做：

YAML
task_id: HOOK-FAILURE-SEMANTICS-FINALIZE-A1

不过我建议它可以放在 live readiness review 之后，根据 readiness review 的结论决定是否作为 blocker 修。
