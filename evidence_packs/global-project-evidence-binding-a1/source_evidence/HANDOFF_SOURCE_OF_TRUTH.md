## Authority hierarchy

### P0 — Canonical evidence sources

P0 sources are the only sources that may establish task verdicts, module status, release/test status, or closure state:

- captured GPT verdicts bound to evidence and verified with `scripts/verify_gpt_reply.py`
- evidence packs under `evidence_packs/**`
- `TEST_OUTPUT` / `TARGETED_TEST_OUTPUT` generated by actual commands
- paper module index: `_reports/PAPER_PROJECT_INDEX.json`
- paper module ledger: `.ai/module_ledger/`
- governance/task ledger: `.ai/tasks/`
- release and test manifests when present

### P1 — GPT-reviewed handoff layer

P1 materials are usable as current handoff inputs only after GPT review:

- GPT-approved `BOOT_CONTEXT.md`
- GPT-approved handoff files
- GPT-approved paste blocks
- handoff approval records and manifests

A coding agent may draft these materials, but cannot declare them final approved.

### P2 — Recall layer

P2 sources are useful for context recovery, not final authority:

- `.claude/skills/claude-memory-compiler/knowledge/**`
- `memory/index.md`
- `memory/knowledge/**`
- `memory/tasks/**`

Memory compiler output may suggest candidate context. Any claim about tests, module status, GPT verdicts, release status, or closure must be checked against P0/P1 sources.

### P3 — Legacy audit/reference layer

P3 sources are retained for history and audit only:

- legacy `PROJECT_HISTORY*`
- legacy `HANDOFF*`
- legacy paste blocks
- root-level `GPT_*.txt` unless verified and explicitly bound in an evidence map

P3 material must not override P0/P1 state.

## Coding-agent handoff role

Coding agents may:

- read local P0/P1/P2/P3 materials
- generate stale checks
- generate handoff drafts
- generate source maps
- generate safety scans
- package evidence packs

Coding agents must not:

- self-approve final handoff materials
- infer `accepted`, `blocked`, or `human_required` from memory or summaries alone
- flatten `accepted_with_limitation` to `accepted`
- flatten `human_required` to `resolved`
- delete, move, rename, or rewrite legacy handoff/history files
- include paper full text, original paragraphs, advisor comments, private notes, or raw transcripts in any handoff artifact

## Paper workflow status rule

Current paper workflow status is:

> Paper workflow is active in bounded, local, privacy-gated mode. Full paper text must not enter GPT conversation or long-term memory.

Older memory entries that describe paper processing as frozen are stale references unless revalidated by P0/P1 evidence.

## Approval rule

Draft materials remain draft until all are true:

1. evidence pack exists,
2. safety scan passes,
3. targeted tests pass,
4. evidence pack linter passes,
5. pre-GPT review gate passes,
6. GPT reply is captured,
7. reply contains `END_OF_GPT_RESPONSE`,
8. `scripts/verify_gpt_reply.py` validates the reply.

Only then may approved handoff artifacts be generated, and only if GPT verdict allows it.
