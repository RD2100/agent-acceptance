需要微调的 5 点
1. runtime-evidence-index.schema.json 不应再写“可选”

你的 TaskSpec 后面已经要求：

并且 acceptance criteria 也要求 runtime negative-path evidence 进入 evidence pack。既然如此，schema 不应写成“可选”。

不要写“可选”。否则执行智能体可能跳过，验收时又会出现“文档要求生成 index，但 schema 没有”的不一致。

2. 把 consistency_check.all_files_agree 从硬编码 true 纳入 A2

这属于 evidence pack 可信度核心问题，建议明确加入 build_evidence_pack.py 改造项：

这个不算扩大范围，因为它直接服务于 “deterministic builder validation”。

3. missing review.yaml / final-report.md 测试要避免循环逻辑

因为 review.yaml 和 final-report.md 是 builder 自己生成的，测试“缺失 review.yaml → needs_more_evidence”时，不能简单在 builder 正常路径里删除它，否则会变成自相矛盾。

建议写清楚：

而不是要求 builder 在正常生成路径中先生成又缺失。

也就是说，A2 最好拆出一个内部函数，例如：

validate_evidence_pack_contract(pack_dir_or_file_list)
classify_evidence_completeness(...)
compute_verdict_eligibility(...)

测试可以对这些函数喂入缺失文件列表。

4. stale_extra_evidence_detected 需要定义判定规则

你列了这个 runtime evidence 场景，但还没有定义什么叫 stale。

建议加一条规则：

stale runtime evidence is detected when:
- scenario header Code version != manifest.head_commit; or
- scenario file generated_at is absent while runtime index requires it; or
- scenario file is present in extra/ but not referenced by runtime-evidence-index.json.

这样 builder 才能确定性判断，不会靠人工猜。

5. --head 参数要明确影响哪些输出

建议写清：

--head must affect:
- manifest.head_commit
- commit_chain
- diff.patch / git diff base..head
- git-show.txt
- runtime-evidence-index Code version comparison

否则只把 head_commit 写进 manifest，但 diff 仍然用 HEAD，就会产生新的证据不一致。
