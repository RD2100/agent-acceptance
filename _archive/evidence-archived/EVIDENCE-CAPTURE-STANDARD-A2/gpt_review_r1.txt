确认：ECS-A2 方向是对的，而且现在应该做。

总体判断

你的 Gate 0 研究是充分的，尤其是三点判断很关键：

1. builder 已经很大，但缺少可验证的 contract；
2. review.yaml / final-report / full regression / runtime evidence 的标准不统一；
3. 测试覆盖偏静态，没有真正验证 evidence pack 质量。

所以 EVIDENCE-CAPTURE-STANDARD-A2 应该立项，而且优先级应为 P0。

我建议将 A2 定义为：

ECS-A2 = canonical evidence pack contract + manifest + verdict eligibility + builder self-validation

而不是：

ECS-A2 = 重写所有证据治理文档 + 修复 SADP review 协议 + 改造旧 evidence pack + 重构 linter

后者会过大。

Q2：verdict_eligibility 应该聚合哪些信号？

你目前计划聚合：

test_pass
conversation_health
startup_read
required_files_present
modified_tracked

这个方向对，但还不够。建议分成 硬阻断信号、降级信号、记录信号 三类。

第一类：硬阻断信号

这些会导致：

verdict_eligibility: needs_more_evidence 或 not_clean_acceptance

建议包括：

YAML
hard_blocking_signals:
  required_files_missing:
    examples:
      - review.yaml
      - final-report.md
      - git-status-after.txt
      - test-output.txt
      - diff.patch or git-show.txt

  modified_tracked_gt_0_without_deferred_record: true

  tests_failed_without_explanation: true

  claimed_full_regression_but_log_missing: true

  runtime_behavior_claimed_but_no_runtime_artifact: true

  safety_error_present:
    examples:
      - secret scan real finding
      - ai-guard error
      - hook blocking stage failed

  schema_invalid_manifest_or_review_yaml: true
第二类：降级信号

这些不一定 blocked，但不能 clean accepted：

YAML
limitation_signals:
  startup_read_missing: limitation
  conversation_health_suggest: limitation
  advisory_stage_warning: limitation
  only_targeted_tests_run: limitation
  runtime_behavior_proved_by_tests_only: limitation
  missing_optional_files: limitation
  ai_guard_warning_explained: limitation

对应结果通常是：

accepted_with_limitation

或者：

needs_more_evidence

取决于任务是否声称相关能力。

第三类：记录信号

这些只进入 manifest，不直接影响 eligibility：

YAML
record_only_signals:
  untracked_total
  session_artifacts
  neg_009_count
  generated_at
  pack_size
  zip_sha256
  warning_count
推荐 verdict_eligibility 枚举

建议不要直接用最终审稿 verdict，而使用“资格状态”：

YAML
verdict_eligibility:
  status: eligible_clean | eligible_with_limitations | needs_more_evidence | not_eligible
  reasons: []
  blocking_signals: []
  limitation_signals: []

这很重要。
verdict_eligibility 不是最终 verdict，它只是告诉 reviewer：

这个 evidence pack 是否具备被 ACCEPTED 的资格。

最终 verdict 仍由 reviewer 判定。

Q6：runtime evidence scenario 格式标准化是否需要 JSON schema？还是保持 .txt + header？
结论：A2 推荐“双轨制”：继续支持 .txt，但新增 machine-readable index。

建议方案：

1. 保持 scenario .txt 文件；
2. 标准化 header；
3. 生成 runtime-evidence-index.json；
4. 为 index 建 schema。
标准 .txt header

例如：

# Scenario: composite_force
# Expected: decision=FORCE_HANDOFF, exit_code=1
# Actual: decision=FORCE_HANDOFF, exit=1
# Status: PASS
# Source: scripts/run_conversation_health_evidence.py
# Generated: 2026-06-12T...
# Code version: <head_commit>
对应 JSON index
JSON
{
  "schema_version": "runtime-evidence-index.v1",
  "scenarios": [
    {
      "name": "composite_force",
      "file": "extra/composite_force.txt",
      "expected": "decision=FORCE_HANDOFF, exit_code=1",
      "actual": "decision=FORCE_HANDOFF, exit=1",
      "status": "PASS",
      "source": "scripts/run_conversation_health_evidence.py",
      "generated_at": "2026-06-12T...",
      "head_commit": "..."
    }
  ]
}

这样兼顾：

人可读：.txt
机器可读：runtime-evidence-index.json
schema 可校验：runtime-evidence-index.schema.json

A2 可以先实现 runtime-evidence-index.json，不必要求每个 scenario 文件本身都是 JSON。
