1. 项目身份与三层架构

本项目是 DevFrame / agent-acceptance / devframe-control-plane 组成的多层 Agent 工作流治理系统。它不是普通代码项目，而是一套围绕 evidence-first、GPT 审查、任务闭环、证据包、审查门禁、上下文压缩 建立的自动化治理框架。

项目核心目标是：让多个编码 agent、GPT 审查、项目蓝本、工作流 runner、论文功能、上下文记忆和 evidence pack 之间形成可复核、可交接、可持续推进的开发流程。

7. 关键教训
7.1 不要把 ready_for_review 当 closed

本项目早期多次出现“测试通过 + 生成报告 + commit/push”就被 agent 视为完成。现在必须坚持：只有 GPT accepted + ledger / PROJECT_HISTORY binding 才能 closed。

7.2 不要提交 summary-only evidence pack

多个任务早期被 blocked，原因是 evidence pack 只有 CLOSURE_REPORT / SAFETY_ATTESTATION / PACK_MANIFEST，没有 actual deliverables。现在每个 evidence pack 必须包含实际代码、schema、测试、diff、manifest 和输出。

7.3 stale evidence 是高频问题

CONTEXT-COMPRESSION-A1 多轮被 blocked，主要原因就是 CLOSURE_REPORT、TARGETED_TEST_OUTPUT、FULL_TEST_OUTPUT 与实际代码不同步。以后必须在最终打包前重新生成所有报告。

7.4 隐私 guard 不能因为“安全文档”就整文件豁免

CONTEXT-COMPRESSION-A1 反复修复了 safety-doc 豁免漏洞。最终规则是：纯 forbidden term list 可以 pass；但只要 marker 后带 payload，就必须 fail closed。

示例：

- raw_paper_text

可以作为安全边界说明 pass。

但：

- raw_paper_text: ACTUAL CONTENT

必须 fail closed。

同理适用于：

YAML
markers:
  - raw_paper_text
  - private_user_text
  - raw_transcript
  - 博士论文正文
  - 用户论文全文
  - api_key
  - token
  - secret
  - cookie
  - session
7.5 dirty worktree split 是当前仓库治理前提

本地 dirty worktree 仍然存在。远程 origin/master 对已 accepted scope 是 clean 的，但本地不是 clean。后续任何 commit 都必须 selected-files-only。
