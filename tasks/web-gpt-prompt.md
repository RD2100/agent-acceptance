# AI 辅助开发治理框架的 6 个结构性弱点 — 深度分析与改造方案

## 背景

我设计了一套 AI 编程代理的治理框架，核心流程是：

```
触发指令 → 前置检查 → 任务规格书 → 执行 → 执行报告 → 独立审计 → 通过/阻断/升级
```

框架包含：P0 硬停规则（禁止危险 git 操作、禁止泄露密钥等）、密钥安全规则、pre-commit 扫描脚本、CI 门禁、GitHub push protection，以及一个可安装的 workflow skill（在编程工具中按需激活）。

## 6 个核心弱点

### 1. 核心悖论：代理自己审计自己

审计器和扫描脚本都由同一个代理触发和运行。即使换了一个模型或 session 来执行审计，dispatch 审计这个动作本身仍然是代理的自主决定。这意味着：
- 代理可以选择不触发审计 → 绕过
- 代理可以伪造执行报告 → 审计器只看报告文本，不验证实际 diff
- 代理可以在审计失败后手动修改判定 → 没有外部不可篡改记录

真正可靠的安全机制需要代理外部的强制执行点。目前已部署 pre-commit hook 和 CI，但代理可以用 `--no-verify` 绕过 hook，可以向不受 CI 保护的分支推送。

**核心问题：代理自愿触发的治理系统能否真正约束代理自身？如果不能，最小可行的外部强制点是什么？**

### 2. 任务规格书是纸老虎

任务规格书要求声明"将要修改的文件列表"，但：
- 实际修改了什么文件没有强制校验——只是事后审计器对比，审计器本身可以被跳过
- 动态生成的文件、临时文件、配置文件不在约束范围内
- "豁免文件"机制是手动维护的白名单，容易膨胀失效

**核心问题：如何在代理执行过程中（而非事后）强制约束文件写入范围？**

### 3. 部署依赖人工

治理框架需要手动 bootstrap 到每个项目，没有自动接入机制。新项目=零保护。

**核心问题：如何让治理规则成为默认生效而非按需安装？**

### 4. 文档密度过高

规范文档 60+ 个文件，大部分是说明文本而非运行代码。代理选择性加载，维护成本高，改了契约要同步多处。

**核心问题：治理框架的"源码"应该是什么？文档、脚本、还是可执行规则？如何精简到最小必要集？**

### 5. 非正式会话零覆盖

日常对话（不触发正式工作流）不经过任何治理检查。代理在普通聊天中执行的任何操作——修改配置、删除文件、写入密钥——完全不可见。

**核心问题：如何在不过度干扰日常对话的前提下，对非正式会话提供底线保护？**

### 6. 审计链单向

审计记录了判定但没有反馈循环：失败后无自动修复、升级后无自动通知人、无累计统计。

**核心问题：审计结果如何驱动后续行为，而不仅仅是记录？**

## 要求

1. 对每个弱点，区分"可以靠代理内部机制解决"和"必须引入外部强制点"
2. 给出具体的改造方案——不是"应该做什么"，而是"怎么实现"
3. 如果某个弱点的根因是同一个问题，合并讨论
4. 给出优先级排序和理由
5. 考虑实现成本和收益的平衡
