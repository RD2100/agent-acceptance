■ OBJECTIVE
Build multi-project isolation architecture: each project gets its own Chrome CDP instance (unique port + isolated profile + dedicated GPT conversation). Enables simultaneous operation of multiple AWSP-governed projects with zero cross-project interference.

■ ARCHITECTURE — TWO CORE MODULES

Module 1: multi_cdp_launcher.py (~310 lines)
- Manages multiple Chrome instances, each on unique port (base_port + index)
- Isolated browser profiles: _cdp_profiles/<project_id>/
- Global registry: .agent/PROJECT_REGISTRY.json
- Functions: _find_chrome, _check_port, _get_version, load_registry, save_registry, launch_chrome, verify_instance
- CLI: launch --projects p1,p2 --base-port 9222 | status | register

Module 2: multi_project_router.py (~270 lines)
- Resolves dispatch targets: project_id → registry → binding → active agent → CDP health
- Functions: load_registry, load_binding, resolve_target, resolve_all, build_dispatch_packet, verify_isolation
- CLI: resolve --project <id> | resolve-all | verify-isolation

■ ISOLATION GUARANTEES (verify_isolation)
- Port uniqueness: no two projects share same CDP debugging port
- Conversation uniqueness: no two projects share same ChatGPT conversation_id
- Profile uniqueness: no two projects share same Chrome browser profile
- Collision detection: all 3 types detected and reported with explicit messages
- Unresolved targets: skipped (not counted in isolation check)

■ DISPATCH FLOW
1. load_registry() → get all registered projects
2. resolve_target(project_id) → registry lookup + binding read + CDP health check
3. build_dispatch_packet(target, task_spec, message) → package for delivery
4. verify_isolation(all_targets) → enforce no cross-project interference
5. Per-project submission script delivers via CDP
