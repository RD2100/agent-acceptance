一、为什么“直接执行”在任务边界处总是失效
因为 agent 的判断逻辑通常是：
YAML当前任务内:  用户已授权: "可以继续改代码、补测试、修 evidence pack"  agent行为: "可以直接执行"当前任务完成后:  状态变成: "一个任务已结束，下一个任务尚未被正式授权"  agent行为: "询问是否继续"
所以“不要再问，直接执行”在 agent 看来只对 A2 当前任务有效，不自动延伸到 CONTROL-PLANE-A1 新任务。
这不是它完全不听话，而是它缺少一个明确的跨任务授权对象。
真正要修的是：GPT 每次发布下一任务时，必须同时给出 next_task_authorization，并明确它就是用户授权。

二、需要固化的机制
以后每个 GPT 审查通过后的回复，都必须包含一个机器可读块：
YAMLnext_task_authorization:  authorized: yes  task_id: "CONTROL-PLANE-A1"  execute_immediately: yes  ask_before_starting: no  allowed_changes:    - "新增/修改文档"    - "新增/修改测试"    - "新增/修改 pipeline 集成代码"    - "新增/修改 CLI / pack validate 调用逻辑"    - "构建 closure evidence pack"  must_stop_and_ask_only_if:    - "删除、移动、重命名、清理 evidence"    - "移除或弱化 guard"    - "启用 live CDP"    - "处理真实用户数据或真实论文"    - "伪造/覆盖历史证据"    - "修改密钥、cookie、session、browser profile"
同时要求 agent 在任务开始前读取这个块。只要 authorized: yes 且 execute_immediately: yes，就不允许再问“继续吗”。
一句话固化：
YAML一句话机制: "每个 accepted 判决后必须附带 next_task_authorization；agent 看到 authorized=yes + execute_immediately=yes 时，直接进入下一任务，不得再次确认。"
