推荐 C 方案：不要补伪 closure，也不要简单排除目录；应当给 runs 目录建立“证据类型分类机制”，让 pre-push governance gate 区分 completed run、historical incomplete run 和 negative test fixture。
这既不删除 evidence，也不伪造 evidence，也不是绕过 hook，而是修正 gate 的错误前提：不是 runs/ 下每个目录都必然代表“已完成闭合的正式 run”。**
YAMLrecommendation: "C：引入 run evidence classification，而不是补假文件或硬编码排除"judgment:  A: "不推荐作为主方案"  B: "不推荐"  C: "推荐"reason: "t-governance-convergence 是历史未闭合 run，negative-incomplete-chain 是负向测试样例；二者都不应被当成正式 completed run 强制要求 closure evidence。"
为什么不推荐 A
A 方案是“补充如实声明未完成闭合的占位文件”。它比伪造 evidence 好，但有一个问题：如果补的是 review.md、review.yaml、final-report.md 这类名字，容易让后续 reviewer 误以为这个 run 已经正式 closure。
可以补“声明文件”，但不应该伪装成正常 closure 文件。也就是说，可以补：
RUN_CLASSIFICATION.yamlINCOMPLETE_RUN_DECLARATION.md
但不建议补：
review.mdreview.yamlfinal-report.md
除非这些文件内部非常明确地写：
YAMLoverall_judgment: review_unverifiedclosure_completed: falseaccepted: false
即便如此，也容易污染证据语义。
为什么不推荐 B
B 方案是“修改 governance gate 排除这两个目录”。这会变成路径级特判，短期能 push，长期会损害 governance gate 的可信度。
问题在于：
这两个目录不是“应该被忽略”，而是“应该被正确分类”。如果直接排除，后续就看不出它们为什么不参与 completed-run 验证。
推荐方案 C：分类验证
把 runs/** 下的 evidence 分成三类：
YAMLrun_evidence_classes:  completed_run:    meaning: "正式闭合 run"    validation: "必须具备完整 closure evidence"  historical_incomplete_run:    meaning: "历史遗留、未完成闭合的 run"    validation: "必须有明确未闭合声明，不允许被计入 accepted evidence"  negative_test_fixture:    meaning: "故意构造的不完整/错误样例，用于测试 gate 能否识别失败"    validation: "必须声明 expected_invalid，不允许被当作正式 run"
这样处理后，pre-push gate 不是被绕过，而是变得更准确。
